1Overview & concept

DukuDukuChat is a single social app that merges a Facebook-style feed, WeChat-style all-in-one messaging and mini-programs, TikTok-style short video, Telegram-style channels and privacy, and Instagram-style visual stories — signed in through the Pi Network (with Google and Apple as alternate sign-in options), and built around the Pi cryptocurrency as its native tipping/payment layer. Its signature differentiator is a single owner-configurable action button at the centre of the navigation bar, letting the app's owner decide in real time what the app's core action is (post, video, live, QR-connect, broadcast, or payment) without shipping a new build.

Why a super-app fits the Pi ecosystem Pi Network's App Platform is built for apps that keep users transacting and engaging inside the Pi ecosystem rather than funnelling them elsewhere. A combined social + payments + mini-programs app is a natural fit, mirroring how WeChat became indispensable in its own ecosystem by combining chat, payments and mini-programs in one place.

2Important: Pi Network listing rule

Before you build, there's one rule from Pi Network's own Mainnet Listing Requirements that directly affects your Google/Apple login request:

Pi apps must use Pi Authentication exclusively to be listed in the Pi App Directory. Pi's Mainnet Listing Requirements state apps must use Pi's authentication system exclusively, keep transactions in Pi, and avoid collecting unnecessary personal data. A version of the app that also offers Google and Apple login can still exist — but it would need to run as a separate, non-Pi-branded web/mobile build outside the Pi App Directory, since only the Pi Browser build can be listed and reviewed by Pi's team.

The recommended approach used throughout this spec is a dual-build strategy:

BuildWhere it runsLogin optionsCan be listed in Pi App Directory?
Pi Browser buildInside the Pi Browser, using the Pi SDKPi Network onlyYes — this is the one you submit to the Pi Developer Portal
Standalone web/mobile buildRegular browsers, App Store, Play StorePi (via Pi SDK web login), Google, AppleNo — distributed independently; can still let Pi users log in with their Pi account

This keeps your product vision (Pi + Google + Apple, one brand) intact while staying eligible for the official Pi listing you asked for.

5Login & identity (Pi / Google / Apple)

Users can create a DukuDukuChat account with any of three identity providers. Accounts can optionally be linked so the same profile is reachable from any provider.

Pi Network sign-in (primary, required for the Pi Browser build)

Uses the official Pi SDK. The app requests scopes, receives an access token, and the backend verifies that token server-side before creating a session.

Client-side (inside Pi Browser)
// Load the Pi SDK script, then:
const scopes = ['username', 'payments'];
function onIncompletePaymentFound(payment) {
  // handle any incomplete payment from a previous session
  console.log('Incomplete payment found', payment);
}

Pi.authenticate(scopes, onIncompletePaymentFound)
  .then((authResult) => {
    // authResult.user.username, authResult.accessToken
    return fetch('/api/auth/pi-verify', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ accessToken: authResult.accessToken })
    });
  })
  .catch((err) => console.error('Pi auth failed', err));
Server-side verification
// Node/Express example
app.post('/api/auth/pi-verify', async (req, res) => {
  const { accessToken } = req.body;
  const me = await fetch('https://api.minepi.com/v2/me', {
    headers: { Authorization: `Bearer ${accessToken}` }
  });
  if (me.status !== 200) return res.status(401).send('Invalid Pi token');
  const piUser = await me.json();
  // create or fetch local user record keyed by piUser.uid, start session
  res.json({ ok: true, user: piUser });
});

Google sign-in (standalone build only)

<script src="https://accounts.google.com/gsi/client" async></script>

<script>
function handleGoogleCredential(response) {
  // response.credential is a signed JWT — verify it server-side
  fetch('/api/auth/google-verify', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ idToken: response.credential })
  });
}
</script>

      body: JSON.stringify({ idToken: res.authorization.id_token })
    });

});

Replace all placeholder client IDs/domains with your own once registered with Pi, Google Cloud Console, and Apple Developer respectively.

6Owner-customisable action button

A single, prominent centre-of-nav-bar button whose icon, colour, label and tap-action are controlled from an owner/admin settings screen — no app update required. This is the app's signature idea, inspired by how WeChat's "+", Telegram's compose button and TikTok's create button each anchor a different core habit.

Example configuration schema

{
  "centerButton": {
    "icon": "camera",              // icon id from the approved icon set
    "color": "#FF5C7A",            // any brand hex colour
    "label": "New Post",           // internal label, shown in the owner dashboard
    "action": "create_post",       // one of the enum values below
    "visibleTo": "all_users"       // all_users | verified_only | creators_only
  }
}

// Allowed "action" values:
// create_post        -> Facebook-style text/photo post composer
// record_video        -> TikTok-style in-app camera for short video
// start_story_or_live  -> Instagram-style story/live capture
// scan_qr_connect      -> WeChat-style QR scanner to add friends / open mini-apps
// new_channel_broadcast -> Telegram-style channel post composer
// open_pi_payment      -> Launch the Pi Wallet send/tip flow
// open_miniapps_menu   -> Open the Mini-Apps marketplace

The owner dashboard should let a non-technical admin pick an icon from a small approved set, a colour swatch, and an action from a dropdown — exactly as demonstrated in the attached clickable prototype (Profile → "Customise centre button"). Changes should be pushed to a remote config store (e.g. Firebase Remote Config, or a simple settings row in your database) and read by the client on app launch, so no app-store update is needed to change it.

9Privacy, safety & moderation

  • Collect only what's needed to run the app — Pi's listing rules specifically flag minimal data collection.
  • Give every post and story an audience selector (Public / Friends / Close Friends / Only Me).
  • Offer optional end-to-end encrypted Secret Chats for sensitive conversations (Telegram-style).
  • Build in blocking, muting and a simple in-app reporting flow from day one — not as a later add-on.
  • Add basic automated content moderation (image/video/text classifiers) before public launch, plus a human review queue.
  • Publish a plain-language privacy policy and terms of service — both are required for Pi Developer Portal review.

10Pi Developer Portal submission checklist

Compiled from Pi Network's public developer documentation. Treat this as your pre-submission gate:

  • Verify your email on your Pi profile, then register the app in the Pi Developer Portal with its name, description and network (start on Testnet).
  • Build and fully test the Testnet version first; only create the Mainnet app entry once it's stable.
  • Host the app at an HTTPS production URL, and verify domain ownership with Pi's required verification file at the domain root.
  • The app's domain must not start with "pi", and must not misuse Pi's logo, colours or branding.
  • The Pi Browser build must use Pi Authentication exclusively — no Google/Apple login inside that build (see section 2).
  • Any in-app financial transactions in the Pi build must be settled in Pi only.
  • Complete KYC verification as the developer before submitting for Mainnet review.
  • Make sure every interactive element works — reviewers reject apps that feel unfinished or broken.
  • Minimise personal data collection and publish a privacy policy.
  • Avoid the app functioning mainly as a funnel out to an external website — keep the core experience inside DukuDukuChat.

Source: Pi Community Developer Guide — Developer Portal, Mainnet Listing Requirements, and Authentication pages (pi-apps.github.io/community-developer-guide, github.com/pi-apps/pi-platform-docs). Requirements can change — re-check the live docs before submitting.

What's real now vs. what still needs you DukuDukuChat_Web.html + server/ is real, working full-stack code — not a simulation. What it can't do on its own: run itself (you need to npm install and start it, or deploy it somewhere with a real HTTPS URL), authenticate with real Pi/Google/Apple credentials (you need to register with each provider and configure their client IDs), or get KYC-verified and listed in the Pi App Directory (that step is tied to your personal identity and has to be done by you through the Pi Developer Portal). DukuDukuChat_App.html remains a simpler, backend-free way to demo the idea without running a server at all.
